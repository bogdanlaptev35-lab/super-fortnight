# Дипломный проект профессии «1C-программист: с нуля до middle»
 
## Кастомизация конфигурации «Управление IT-фирмой для компании «Ваш компьютерный мастер»

В рамках проекта автоматизированы две ключевые подсистемы:
1. **Работа с заявками клиентов** — планирование, фиксация выполнения и последующий учет работ.
2. **Расчёт управленческой зарплаты** — учет выработки специалистов, расчет окладов, премий, отпускных и ведение взаиморасчетов.

---

##  1. Начальная настройка системы

### 1.1. Интеграция с Telegram-ботом оповещений
Для оперативного информирования специалистов о распределении новых заявок в системе используется интеграция с мессенджером Telegram.

1. **Регистрация бота:**
   - Найдите в Telegram официального бота `@BotFather`.
   - Отправьте команду `/newbot` и следуйте инструкциям для задания понятного имени и уникального юзернейма (должен заканчиваться на `bot`).
   - Скопируйте сгенерированный токен API (вид: `123456789:ABCdefGhIJKlmNoPQRsTUVwxyZ`).
2. **Внесение настроек в 1С:**
   - Запустите 1С в пользовательском режиме.
   - Перейдите к редактированию констант и вставьте скопированный токен в константу **«Токен управления телеграм-ботом»**.
3. **Определение идентификатора группы (Chat ID):**
   - Создайте рабочую группу в Telegram, добавьте туда вашего бота и назначьте его администратором группы.
   - Напишите любое текстовое сообщение в созданную группу.
   - Откройте браузер и перейдите по адресу: `https://api.telegram.org/bot<ВашТокенБот>/getUpdates`.
   - В полученном ответе в формате JSON найдите объект `"chat"` и скопируйте значение поля `"id"` (обычно начинается с минуса для групп, например `-100123456789`).
   - Вставьте полученное значение в константу **«Идентификатор группы для оповещения»** в 1С.
4. **Регламентные задания:**
   - Откройте консоль регламентных заданий и включите расписание для регламентного задания **«Отправка уведомлений телеграм боту»** для автоматической отправки накопленной очереди сообщений.

### 1.2. Подготовка базы для автоматизированного тестирования
Все ключевые пользовательские сценарии покрыты автотестами на базе фреймворка **Vanessa Automation**.

1. **Развертывание и параметры запуска:**
   - Создайте чистую локальную копию информационной базы.
   - **Важно:** При первом запуске или переносе базы в диалоговом окне обязательно выберите вариант **«База является перенесенной»**, чтобы предотвратить блокировку регламентных заданий и внешних соединений.
   - В настройках запуска информационной базы в окне запуска 1С (кнопка «Изменить» -> далее -> далее) в поле *«Дополнительные параметры запуска»* пропишите ключ: `/TESTMANAGER`.
2. **Настройка безопасности:**
   - Зайдите в базу под учетной записью с правами администратора.
   - Перейдите в раздел *Администрирование -> Настройки пользователей и прав -> Пользователи*.
   - Выберите пользователя `Администратор`, откройте его карточку и снимите флаг **«Безопасный режим»** (требуется для корректного выполнения внешних обработок фреймворком тестирования).

##  2. Запуск автотестов (Vanessa Automation)

Файлы пользовательских сценариев (`.feature`) расположены в корневом каталоге репозитория.

1. Скачайте актуальную версию фреймворка `Vanessa Automation` (файл `vanessa-automation.epf`) из официального репозитория разработчиков и сохраните локально.
2. Запустите 1С в режиме Менеджера тестирования (с установленным флагом `/TESTMANAGER`).
3. Откройте внешнюю обработку через *Главное меню -> Файл -> Открыть -> vanessa-automation.epf*.
4. **Конфигурирование клиентов тестирования:**
   - Перейдите на вкладку **«Клиенты тестирования»**.
   - Добавьте три записи со следующими параметрами для имитации многопользовательской работы:
     
     | Имя клиента | Тип | Дополнительные параметры запуска |
     | :--- | :--- | :--- |
     | **Менеджер** | Тонкий/Толстый клиент | `/N"ИмяПользователяМенеджера"` |
     | **Специалист** | Тонкий/Толстый клиент | `/N"ИмяПользователяСпециалиста"` |
     | **БухгалтерИТФирмы** | Тонкий/Толстый клиент | `/N"ИмяПользователяБухгалтера"` |

5. **Загрузка и выполнение сценариев:**
   - Перейдите обратно на вкладку **«Запуск сценариев»**.
   - Нажмите кнопку **«Открыть feature файл...»** или выберите команду меню **«Загрузить один файл»**.
   - Выберите один из предзагруженных сценариев из репозитория:
     - `Менеджер.feature` — создание документов «Обслуживание клиента», проверка отправки сообщений в Telegram.
     - `Специалист.feature` — закрытие и проведение документов «Обслуживание клиента» с заполнением ТЧ выполненных работ.
     - `БухгалтерИТФирмыМассовое.feature` — запуск обработки группового формирования актов.
     - `БухгалтерИТФирмыАнализ.feature` — автоматическое формирование отчета «Анализ выставленных актов» и проверка показателей.
   - Нажмите кнопку **«Запустить сценарии на выполнение»** (иконка зеленого треугольника Play). Если учетные записи содержат пароли, введите их в появившемся окне авторизации.

##  3. Архитектура изменений и стандарты разработки

### 3.1. Структура метаданных и подсистем
Все новые объекты конфигурации изолированы и объединены в специализированную служебную подсистему **«ВКМ Добавленные Объекты»**. 

Для корневых объектов метаданных (документы, справочники, журналы, регистры) применяется обязательный уникальный префикс **`ВКМ_`** (например, документ `ВКМ_ОбслуживаниеКлиента`, документ `ВКМ_НачисленияЗарплаты`).

### 3.2. Стандарты именования внутренних реквизитов (Правило чистоты объектов)
В соответствии со стандартами разработки БСП и требованиями к чистоте кода:
- **Внешний уровень:** Префикс `ВКМ_` используется исключительно для самих объектов конфигурации верхнего уровня. Это позволяет визуально отделить кастомные сущности от объектов поставщика.
- **Внутренний уровень:** Внутри объектов с префиксом `ВКМ_` (в составе их реквизитов, табличных частей, макетов и форм) использование префикса **запрещено**. Реквизиты должны именоваться естественно (например: `Клиент`, `Договор`, `Специалист`, `СуммаКОплате`), поскольку контекст принадлежности к кастомному объекту уже очевиден.

### 3.3. Программное управление доступом
Новые роли и поставляемые профили групп доступа описываются строго программным способом в общем модуле **`УправлениеДоступомПереопределяемый`** в процедуре `ПриЗаполненииПоставляемыхПрофилейГруппДоступа`. 

Основные профили прав:
- **Менеджер IT-фирмы** — создание заявок, оперативное планирование.
- **Специалист IT-фирмы** — просмотр назначенных задач, заполнение отчетов о фактически потраченных часах.
- **Бухгалтер IT-фирмы** — регламентный учет, массовое формирование документов закрытия периода.
- **Кадровик-расчетчик** — ведение кадровых приказов, расчет управленческой зарплаты, премий и графиков отпусков.

### 3.4. Оптимизация работы с транзакциями
При разработке алгоритмов массового создания документов (например, в обработке «Массовое создание актов») **не допускается** избыточное явное оборачивание записи одиночных объектов в транзакции (`НачатьТранзакцию() ... ЗафиксироватьТранзакцию()`). 

Платформа 1С:Предприятие гарантирует атомарность записи на уровне метода `Объект.Записать()`. Использование дополнительных явных транзакций при записи одного объекта перегружает менеджер блокировок СУБД и снижает общую производительность системы. Перехват возможных ошибок записи должен осуществляться через стандартный блок `Попытка...Исключение`.

##  4. Функциональный состав блоков

### Подсистема «Работа с заявками клиентов»
- **Договоры на абонентское обслуживание:** Реализован расширенный учет параметров договора (период действия, фиксированная сумма ежемесячной абонентской платы, лимит стоимости часа специалиста).
- **Обслуживание клиентов:** Документ, фиксирующий весь цикл работы по заявке — от фиксации проблемы менеджером до отражения фактически выполненных работ и потраченных часов специалистом.
- **Отчет «Анализ выставленных актов»:** Сравнительный отчет, сопоставляющий плановые объемы работ по договорам обслуживания с фактически выставленными за месяц актами по номенклатуре.

### Подсистема «Расчёт управленческой зарплаты»
- **Начисление и Выплата зарплаты:** Механизмы расчета фиксированной и переменной (процент от выработки) частей оплаты труда.
- **Учет отпусков:** Документы планирования и распределения отпусков сотрудников с контролем остатков дней.
- **Отчет «Расчёты с сотрудниками»:** Отражает начисления, выплаты и конечное сальдо взаиморасчетов с персоналом.
- **Отчет «Расход запланированных отпусков»:** Годовая аналитика соответствия фактических отпусков утвержденному плану графиков.
